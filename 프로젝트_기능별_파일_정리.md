## 🔗 전체 인증 흐름

```
[프론트엔드]
AppRouter.jsx (라우팅 진입)
    ↓
PrivateRouter.jsx (PrivateRoute) → AuthContext.jsx(isAuthenticated 확인)
    ↓
axios.jsx (요청 인터셉터: Authorization 헤더에 JWT 자동 첨부)
    ↓
chatApi.js / weather.js / authApi.js 등 각 API 파일
    ↓
[백엔드]
JwtAuthenticationFilter.java (SecurityConfig에 등록된 실제 필터)
    └→ JWTUtil.java (토큰 검증·파싱 실행)
    ↓
SecurityConfig.java (인증된 사용자만 접근 가능한 경로 제어)
    ↓
Controller → Service → DAO/Repository → DB
```

> **학습 시작점 추천**
> - 백엔드 인증 흐름: `JWTUtil.java` → `JwtAuthenticationFilter.java` → `SecurityConfig.java` → `MemberController.java`
> - 프론트 인증 흐름: `AppRouter.jsx` → `PrivateRouter.jsx` → `AuthContext.jsx` → `axios.jsx`
> - 채팅: `WebSocketConfig.java` → `ChatController.java` → `ChatContext.jsx` → `ChatRoomDetail.jsx`
> - 날씨: `WeatherController.java` → `WeatherService.java` → `weather.js`
> - 뉴스: `GlobalEcoNewsController.java` → `GlobalEcoNewsService.java` → `GlobalEcoNews.jsx`

## 🗑️ 중복·미사용 설정 파일 분석

> 코드를 직접 grep으로 검색해서 확인한 결과입니다.

---

### ❌ 1. `config/JwtFilter.java` — **완전 미사용 (데드코드)**

**검증 방법:** 프로젝트 전체에서 `import com.kh.spring.config.JwtFilter` 검색 → **결과 없음**

```java
// SecurityConfig.java 에 실제 등록된 것
.addFilterBefore(jwtAuthenticationFilter, ...);  // ← util/JwtAuthenticationFilter 사용

// config/JwtFilter.java 는 아무도 참조하지 않음
```

두 파일 모두 `JWTUtil.java`를 써서 동일한 JWT 검증 로직을 가지지만, 단 하나의 차이가 있습니다:

| 파일 | 권한 설정 |
|------|----------|
| `config/JwtFilter.java` | 권한 빈 리스트 `Collections.emptyList()` |
| `util/JwtAuthenticationFilter.java` ✅ | `ROLE_USER` 권한 부여 |

**결론:** `config/JwtFilter.java`는 삭제해도 아무 영향 없는 잔여 코드입니다.

---

### ⚠️ 2. CORS 설정이 두 군데 중복

| 파일 | CORS 설정 방식 | 실제 적용 여부 |
|------|---------------|--------------|
| `config/WebConfig.java` | `addCorsMappings()` — Spring MVC 레벨 | ⚠️ Security 통과 후 적용 (후순위) |
| `common/config/SecurityConfig.java` ✅ | `corsConfigurationSource()` — Security 레벨 | ✅ 먼저 적용됨 |

**왜 중복인가?**
Spring Security가 모든 HTTP 요청을 MVC보다 먼저 처리합니다. Security 단계에서 CORS를 처리하면 MVC 단계의 CORS 설정은 사실상 무의미합니다.

**단, WebConfig는 완전 삭제 불가:** Swagger 리소스 핸들링(`/swagger-ui/**`, `/webjars/**`)과 정적 리소스 설정도 함께 들어있기 때문에, **CORS 부분만 중복**입니다.

---

### ⚠️ 3. `config/RestTemplateConfig.java` — **Bean 등록만 하고 아무도 주입받지 않음**

**검증 방법:** 전체 코드에서 `@Autowired RestTemplate` 또는 생성자 주입으로 `RestTemplate`을 받는 곳 검색 → **없음**

실제로 RestTemplate을 쓰는 파일들을 보면 모두 직접 생성합니다:

```java
// GlobalEcoNewsService.java
RestTemplate restTemplate = new RestTemplate();  // ← new로 직접 생성

// KakaoService.java
RestTemplate rt = new RestTemplate();  // ← new로 직접 생성

// map/service/SeoulMapClient.java
private final RestTemplate restTemplate = new RestTemplate();  // ← 필드에서 직접 생성
```

**결론:** Bean으로 관리되는 RestTemplate이 있는데, 아무도 그 Bean을 쓰지 않고 각자 `new RestTemplate()`으로 만들어 씁니다. Bean의 장점(싱글톤 관리, 테스트 용이성 등)을 못 살리고 있는 코드 구조입니다.

---

### ⚠️ 4. `config/WebClientConfig.java` — **Bean 설정이 실제로 적용 안 됨**

**검증 방법:** `WeatherService.java` 70번째 줄 직접 확인

```java
// WeatherService.java - Config Bean 주입 없이 직접 생성
WebClient webClient = WebClient.builder()
        .baseUrl("https://apis.data.go.kr/...")
        .build();  // ← WebClientConfig의 5MB 설정이 적용 안 됨!
```

`WebClientConfig`에서는 5MB 메모리 한도를 설정해뒀는데:
```java
// WebClientConfig.java
.codecs(c -> c.defaultCodecs().maxInMemorySize(5 * 1024 * 1024))
```

`WeatherService`가 이 Bean을 주입받지 않고 `WebClient.builder()` 직접 사용 → 기본 256KB 메모리 한도로 동작 중입니다. 대용량 날씨 응답이 오면 에러가 날 수 있습니다.

---

### ✅ 5. `ChatWebConfig` / `CommunityWebConfig` / `WebConfig` — 중복 아님, 역할 분리 정상

| 파일 | 담당 URL | 역할 |
|------|---------|------|
| `ChatWebConfig.java` | `/chat/file/**` | 업로드된 채팅 파일을 클라이언트에 서빙 |

| `WebConfig.java` | `/swagger-ui/**`, `/**` | Swagger UI 리소스 + 정적 파일 핸들링 |

서로 다른 URL 패턴을 처리하는 것들이라 **각자 존재 이유가 있습니다.**

### 🗺️ 추천 공부 순서 (Why?)

```
1단계 (기반) → 2단계 (인증 흐름) → 3단계 (채팅) → 4단계 (날씨) → 5단계 (뉴스) → 6단계 (설정/심화)
```

- **1단계부터 시작하는 이유:** JWT·Security·Axios 인터셉터를 모르면 다른 기능 코드를 읽어도 "왜 이 헤더가 붙지?", "왜 이 경로는 막히지?" 이해 불가
- **채팅을 먼저 보는 이유:** WebSocket·JPA·이벤트 시스템 등 가장 복잡한 패턴이 집약되어 있어, 이걸 이해하면 날씨·뉴스는 훨씬 쉽게 보임
- **날씨·뉴스를 나중에 보는 이유:** REST 호출 + 캐시 패턴만 알면 끝나서 상대적으로 단순함
