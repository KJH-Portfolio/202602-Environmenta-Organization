## 💡 문제 해결 및 기술 철학 (Technical Note)
- **데이터 무결성 최우선**: 단순한 기능 구현보다 데이터의 일관성(Consistency)을 최우선으로 하며, 예외 상황에서도 데이터가 오염되지 않는 **방어적 아키텍처**를 지향합니다.
- **성능과 비용의 균형**: AI 및 외부 API 연동 시 발생하는 리소스 소모를 최소화하기 위해 **비동기 처리(Non-blocking)**와 **캐싱 전략**을 적극 활용합니다.
- **근본 원인 분석(Root Cause Analysis)**: 로그 분석과 실행 계획 확인을 통해 일시적인 해결이 아닌 근본적인 시스템 개선을 추구합니다.

## 💎 Case 1: 분산된 데이터의 원자적 정합성 보장 (Transaction)

### 🚩 Situation
- 사용자가 환경 활동(에코 맵 경로 주행)을 완료했을 때, **[탄소 기록 저장]**, **[포인트 지급]**, **[에코트리 경험치 상승]**이라는 3가지 상이한 도메인의 DB 작업이 동시에 발생함.
- 네트워크 오류나 서버 문제로 인해 일부 작업만 성공하고 나머지는 실패할 경우, 사용자의 포인트와 실제 활동 기록이 불일치하는 심각한 정합성 결함 발생 위험 식별.

### 🔍 Cause
- 각 도메인이 별도의 서비스 레이어로 분리되어 있어, 호출 순서에 따라 중간에 예외가 발생할 경우 이전 작업이 롤백되지 않는 구조적 한계.

### ✅ Solution
- **@Transactional의 선언적 트랜잭션 적용**: 비즈니스 로직을 담당하는 통합 서비스 메서드에 `@Transactional`을 적용하여 **All-or-Nothing** 원칙을 강제함.
- **Custom Exception 정의**: 특정 비즈니스 조건 미달 시 의도적인 `RuntimeException`을 발생시켜 전체 프로세스를 안전하게 롤백하고 사용자에게 명확한 에러 메시지를 반환하도록 설계함.

### 📈 Result
- 데이터 불일치 발생 건수 **0건** 달성.
- 복잡한 연쇄 작업 중 장애 발생 시에도 데이터 무결성을 100% 보장하는 견고한 백엔드 시스템 구축.

## 🔥 Case 2: AI 연동 시의 응답 지연 및 병목 현상 (Performance)

### 🚩 Situation
- Google Gemini AI와 통신하여 환경 조언을 생성하는 과정에서 외부 API의 응답 속도(평균 2~4초)가 길어짐에 따라, 전체 API 서버의 쓰레드가 점유되어 다른 일반 요청들의 처리 속도까지 저하되는 현상 발생.

### 🔍 Cause
- 기존 `RestTemplate`의 **Blocking I/O** 방식은 외부 서버로부터 응답이 올 때까지 쓰레드가 대기 상태로 머물러 있어 리소스 낭비가 심함.

### ✅ Solution
- **Spring WebClient 도입**: Non-blocking 방식의 `WebClient`를 활용하여 AI 요청을 비동기적으로 처리. 요청을 보낸 후 응답을 기다리지 않고 쓰레드를 반환하여 다른 작업을 수행하도록 개선.
- **Caffeine Local Cache 적용**: 동일한 환경 키워드에 대한 AI 답변은 1시간 동안 로컬 캐시에 저장하여, 불필요한 외부 API 호출을 80% 이상 감소시킴.

### 📈 Result
- 평균 응답 시간(Latency) **약 75% 개선**.
- API 호출 비용 절감 및 대규모 트래픽 상황에서도 안정적인 AI 서비스 제공 가능.

## 💬 Case 3: 실시간 통신 세션 유실 및 상태 동기화 (WebSocket)

### 🚩 Situation
- 실시간 채팅 및 알림 서비스 운영 중, 불안정한 네트워크 환경에서 WebSocket 연결이 유실되었을 때 사용자의 '접속 중' 상태가 DB와 동기화되지 않아 오보가 발생하는 문제 발생.

### 🔍 Cause
- 세션 종료 이벤트가 정상적으로 서버에 도달하지 못하는 경우, 서버는 해당 세션을 여전히 활성 상태로 인지하여 메모리 누수 및 잘못된 상태 값 유지.

### ✅ Solution
- **STOMP Interceptor & Heartbeat**: `ChannelInterceptor`를 구현하여 클라이언트의 연결/해제 이벤트를 감시하고, 주기적인 **Heartbeat** 패킷을 통해 유령 세션을 감지하여 즉시 DB 상태를 `OFFLINE`으로 갱신하는 로직 구현.
- **SockJS Fallback**: WebSocket 미지원 환경이나 연결 불안정 시 자동으로 HTTP Long Polling으로 전환하는 Fallback 메커니즘 적용.

### 📈 Result
- 사용자 접속 상태의 실시간 정확도 **99% 이상** 확보.
- 세션 관리 최적화를 통해 불필요한 서버 메모리 점유율을 30% 감소시킴.
